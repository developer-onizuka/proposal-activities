# proposal-activities

# Goal
生成AI向けAIサーバーを提案・設計するために必要になってくるであろう並列化技術、KVキャッシュ、IO技術、量子化や、Day2オペレーションの実践的な知識を理解する。

# 第1章：【要件定義】ビジネス要件をインフラ仕様へ変換するワークロード分析

### サマリー
生成AIインフラの設計において最も重要な最初の意思決定は、顧客の抽象的なビジネス要件を、物理的なインフラ仕様へと論理的に翻訳するプロセスである。この設計は、単に最新のGPUを並べることではなく、タスクの複雑性、精度、コストの三原則から逆算される収束プロセスとして定義される。

### 1. タスクの難易度によるモデルサイズの絞り込み
まず、解決すべきビジネス課題の複雑性から、必要十分な知能（パラメータ数）を特定する。定型的な要約や単純な抽出タスクであれば8B前後の軽量モデルで十分であるが、高度な論理推論やプログラミング支援が必要な場合は70Bクラス以上の中・重量級モデルが選択肢に入る。ここで重要なのは、知能のオーバースペックを避けることである。パラメータ数が増えるほど、後述するインフラコストとレイテンシが指数関数的に増大するためである。

### 2. 物理制約：実効VRAM容量の算出
モデルが決まれば、物理的なメモリ要件を算出する。FP16精度（2バイト/パラメータ）を基準とした場合、モデルの重みだけで **パラメータ数 × 2バイト** のVRAMを占有する。しかし、実務設計で肝となるのは、入力トークン長に比例して肥大化するKVキャッシュ分の動的バッファを加味した実効メモリの計算である。特にRAG（外部検索）などで長大なコンテキストを扱う場合、モデル自体よりもこのキャッシュ領域がボトルネックとなり、GPUの枚数やメモリ帯域（HBM）への要求を決定づける。

### 3. パフォーマンス指標とGPUの選定ロジック
最終的なデバイス選定は、目標とするトークン生成速度とTCO（総保有コスト）の天秤で決まる。

* **推論のスループット重視**: コスト効率を優先し、中規模以下のモデルを量子化して運用する場合、L4やL40SといったシングルスロットGPUを並列化する構成が有効である。
* **学習および超低レイテンシ推論**: 大規模LLMの学習や、極めて高い応答速度が求められる商用環境では、高帯域メモリを備え、GPU間を高速なNVLinkで結合できるA100やH100/H200が必然の選択となる。



このように、ビジネス課題から必要な知能（モデル）、そして物理実効メモリの制約を経て性能限界（GPU種別）へと至る論理の鎖を構築することで、単なるスペック提案ではない、経営と技術を繋ぐ必然性の高いアーキテクチャが完成する。

---
実際の提案現場で想定される判断のプロセスを例示する。

#### Q1: 特定のAIワークロードに対して、適切なGPUタイプと数をどのように決定するか？

**回答の論理構成例：**
例えば、Llama-3 70Bモデルを用いた社内ドキュメント検索（RAG）システムを構築する場合、以下のステップでモデルとインフラを絞り込む。

1. **PoCによる精度検証**
   まずは8Bと70Bの両モデルで精度比較を行い、業務要件を満たす最低ラインのパラメータ数を特定する。仮に70Bが必要と判断された場合、FP16精度ではモデル単体で140GB、KVキャッシュを含めると160GB以上のVRAMが必要となる。
2. **インフラの試算とリソース提示**
   目標応答速度が1秒間に20トークンであれば、メモリ帯域（HBM）が重要になる。80GBのVRAMを持つH100を2枚、NVLinkで接続する構成を基本案として提示する。
3. **コスト対精度の代替案提示**
   予算が限られている場合は、モデルの量子化を提案する。70Bを量子化してVRAMを圧縮しH100を1枚に抑える案や、あるいは「量子化した8BモデルをL4で回せば、精度は若干落ちるもののコストは1/5に抑えられる」といった比較案を提示し、ビジネス価値とコストのバランスに基づく意思決定を支援する。




# 第2章：【基本設計】投資対効果（ROI）を最大化するスループットとコストの最適化
論理: 限られた予算内で最大のスループットを出すために、バッチサイズとレイテンシのトレードオフをどう設計に組み込むか。

核心: SLA遵守とコスト効率を両立させるための、ルーフライン・モデルに基づく理論限界の把握。

# 第3章：【詳細設計】ソフトウェアでハードウェアの限界を超える：推論エンジンの戦略的採択
論理: 単に載せるだけでは性能は出ない。vLLMやTritonが必要なのは、ハードウェアのアイドル時間を物理的にゼロに近づけるため。

核心: PagedAttentionや動的バッチングを「必然の解決策」として組み込むアーキテクチャ設計。

# 第4章：【実装・最適化】推論の死角を排除する：メモリ管理とI/Oパイプラインの設計
論理: 理論上は動くはずのモデルが、なぜ現場で止まるのか。OOMやGPU Starvationを未然に防ぐための、メモリとデータの「流れ」の設計。

核心: KVキャッシュの動的挙動の予測と、前処理・転送・演算のオーバーラップによるボトルネック解消。

# 第5章：【コスト最適化】精度を損なわずにTCOを下げる：量子化技術の導入判断
論理: スケールするにつれて膨らむインフラコストへの処方箋。精度維持と計算効率向上のバランスをどう評価し、顧客に提案するか。

核心: 量子化手法（AWQ/GPTQ等）の選択が、将来的な推論コストとハードウェア寿命に与えるインパクト。

# 第6章：【拡張設計】スケールアップへの備え：マルチノード・クラスタと高速インターコネクト
論理: 単一ノードの限界を迎え、ノードを跨ぐ際の「通信の壁」をどう突破するか。RDMAやNCCLが必要になる物理的な必然性。

核心: 分散環境におけるデータ転送の最適化と、スケーラビリティを担保するネットワーク・トポロジー。

# 第7章：【デプロイ】Cloud Nativeな機動力の実装：GKEによる自動化と弾力性の確保
論理: 変化の速いAIサービスにおいて、いかに迅速にデプロイし、需要に応じてリソースを伸縮（オートスケール）させるか。

核心: コールドスタート対策とスポットインスタンス活用による、実運用レベルでのコスト・デリバリー最適化。

# 第8章：【Day 2 運用】「定常」を維持する信頼性工学：監視・障害対応・継続的改善
論理: 構築はスタートに過ぎない。GPUの健康状態を監視し、サイレント故障によるサービスダウンを回避するための運用フレームワーク。

核心: DCGMによる予兆検知と、パフォーマンス劣化を許さない継続的なキャパシティ・プランニング。

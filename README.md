# proposal-activities

# Goal
生成AI向けAIサーバーを提案・設計するために必要になってくるであろう並列化技術、KVキャッシュ、IO技術、量子化や、Day2オペレーションの実践的な知識を理解する。

# 第1章：【要件定義】ビジネス要件をインフラ仕様へ変換するワークロード分析

### サマリー
生成AIインフラの設計において最も重要な最初の意思決定は、顧客の抽象的なビジネス要件を、物理的なインフラ仕様へと論理的に翻訳するプロセスである。この設計は、単に最新のGPUを並べることではなく、タスクの複雑性、精度、コストの三原則から逆算される収束プロセスとして定義される。

### 1. タスクの難易度によるモデルサイズの絞り込み
まず、解決すべきビジネス課題の複雑性から、必要十分な知能（パラメータ数）を特定する。定型的な要約や単純な抽出タスクであれば8B前後の軽量モデルで十分であるが、高度な論理推論やプログラミング支援が必要な場合は70Bクラス以上の中・重量級モデルが選択肢に入る。ここで重要なのは、知能のオーバースペックを避けることである。パラメータ数が増えるほど、後述するインフラコストとレイテンシが指数関数的に増大するためである。

### 2. 物理制約：実効GPUメモリ容量の算出
モデルが決まれば、物理的なメモリ要件を算出する。FP16精度（2バイト/パラメータ）を基準とした場合、モデルの重みだけで **パラメータ数 × 2バイト** のGPUメモリを占有する。しかし、実務設計で重要なのは、入力トークン長に比例して肥大化するKVキャッシュ分の動的バッファを加味した実効メモリの計算である。特にRAG（外部検索）などで長大なコンテキストを扱う場合、モデル自体よりもこのキャッシュ領域がボトルネックとなり、GPUの枚数やメモリ帯域（HBM）への要求を決定づける。

### 3. パフォーマンス指標とGPUの選定ロジック
最終的なデバイス選定は、目標とするトークン生成速度とTCO（総保有コスト）のバランスで決まる。

* **推論のスループット重視**: コスト効率を優先し、中規模以下のモデルを量子化して運用する場合、L4やL40SといったシングルスロットGPUを並列化する構成が有効である。
* **学習および超低レイテンシ推論**: 大規模LLMの学習や、極めて高い応答速度が求められる商用環境では、高帯域メモリを備え、GPU間を高速なNVLinkで結合できるA100やH100/H200が必然の選択となる。



このように、ビジネス課題から必要な知能（モデル）、そして物理実効メモリの制約を経て性能限界（GPU種別）へと至るロジックを十分検討することで、単なるスペック提案ではない経営と技術を繋ぐ必然性の高いアーキテクチャが完成する。

---
実際の提案現場で想定される判断のプロセスを例示する。

#### Q1: 特定のAIワークロードに対して、適切なGPUタイプと数をどのように決定するか？

**回答の論理構成例：**
例えば、Llama-3 70Bモデルを用いた社内ドキュメント検索（RAG）システムを構築する場合、以下のステップでモデルとインフラを絞り込む。

1. **PoCによる精度検証**
   まずは8Bと70Bの両モデルで精度比較を行い、業務要件を満たす最低ラインのパラメータ数を特定する。仮に70Bが必要と判断された場合、FP16精度ではモデル単体で140GB、KVキャッシュを含めると160GB以上のGPUメモリが必要となる。
2. **インフラの試算とリソース提示**
   目標応答速度が1秒間に20トークンであれば、メモリ帯域（HBM）が重要になる。80GBのGPUメモリを持つH100を2枚、NVLinkで接続する構成を基本案として提示する。
3. **コスト対精度の代替案提示**
   予算が限られている場合は、モデルの量子化を提案する。70Bを量子化してGPUメモリを圧縮しH100を1枚に抑える案や、あるいは「量子化した8BモデルをL4で回せば、精度は若干落ちるもののコストは1/5に抑えられる」といった比較案を提示し、ビジネス価値とコストのバランスに基づく意思決定を支援する。

# 第2章：【基本設計】投資対効果（ROI）を最大化するスループットとコストの最適化

### サマリー
本章の目的は、第1章で選定したデバイス性能を最大限に引き出し、投資対効果（ROI）を確定させることにある。生成AIインフラの運用コストを最適化する鍵は、計算リソースが遊ばないように配慮しつつ、大量のトークンを処理するスループットを向上させる点にある。しかし、スループットの追求はユーザーの体感速度であるレイテンシとの間に強力なトレードオフを生じさせる。そのため、サービス品質保証（SLA）を遵守しつつ、計算リソースの性能限界を理論的に把握することで、限られた予算内でコスト効率を両立させる最適解を設計に組み込むプロセスを解説する。

---

### 1. バッチサイズによる性能変化のメカニズム
GPUの性能を評価する際、単なる演算理論値だけでなく、メモリからデータを読み出す速度と演算器が計算する速度のどちらがボトルネックになっているかを見極める必要がある。
小規模なバッチサイズ、あるいは1リクエストずつの逐次処理では、演算器の処理能力に対してデータの供給が追いつかないメモリ転送速度の限界に直面する。この状態では、どれほど高価なGPUを導入しても、演算リソースの大部分が待機状態となってリソースが遊んでしまい、コスト効率が著しく低下する。

### 2. スループットとレイテンシの関係
バッチサイズ、つまり一度に並列処理するリクエスト数を拡大すると、1回のデータ読み出しに対して計算密度が高まり、GPUの演算効率は飛躍的に向上する。これがスループットの最大化をもたらす。
しかし、バッチサイズを大きくしすぎると、計算自体の完了を待つ時間が増え、最初の1文字目が出るまでの時間（TTFT）や、全体の応答時間が悪化する。ここで重要となるのが、ビジネス側が設定したSLA（応答速度の許容範囲）である。インフラ設計者は、このSLAの境界線を超えない範囲で、計算リソースの理論的な性能飽和点を狙ってバッチサイズを調整しなければならない。

### 3. コスト効率を両立させる最適動作点の特定
最終的な設計指針は、ハードウェアが持つ計算性能の限界を可視化して把握することである。演算性能が飽和する直前、かつSLAを維持できるポイントこそが、最もトークンあたりの単価が安くなる動作点となる。
このように、ハードウェアの物理的な特性とユーザー体験のバランスを意識し、理論的な裏付けを持って最も効率的に稼働する設定を導き出すことが重要である。



---
性能設計の現場で問われる判断のプロセスを例示する。

#### Q2: 計算リソースが遊ばないように効率を最大化しつつ、応答速度を維持するための具体的なアプローチは何か？

**回答の論理構成例：**
例えば、大規模な顧客基盤を持つ企業のチャットサービスを運用する場合、単一のGPU性能だけでなく、リソース全体の断片化を防ぐ視点が不可欠となる。

1. **性能特性の把握とSLAの定義**
   使用するGPUにおいて、メモリ帯域がボトルネックとなっている領域を特定する。同時に、ビジネス要件として応答速度3秒以内というSLAを定義し、それを維持できる限界のバッチサイズを測定する。
2. **ソフトウェアによるリソースの断片化解消**
   個別のリクエストを単純に各GPUへ割り振ると、あるGPUは高負荷でレイテンシが悪化し、別のGPUはリソースが余るといった断片化が生じる。これを防ぐため、リクエストを集中管理し、各リソースの稼働状況に合わせて配分するオーケストレーションを導入する。
   また、**PagedAttention**を導入し、OSの仮想メモリのようにVRAMをブロック単位で動的に管理することで、KVキャッシュによるメモリの内部断片化を排除する。これにより、メモリ利用率を極限まで高め、1枚のGPUで処理可能な同時リクエスト数を最大化させる。
   
3. **継続的バッチング（Continuous Batching）による効率化**
   従来の静的バッチングでは、最も長い回答の完了を待つ間に他の演算リソースが遊んでしまう課題があった。これに対し、推論エンジン側でリクエストの完了を待たずに新たなリクエストを順次詰め込む**継続的バッチング**を採用する。
   
   これにより、GPU内の演算器が遊ぶ時間をゼロに近づけ、SLAを遵守しながらトークンあたりの処理コストを最小化、投資対効果を最大化させる。

# 第3章：【詳細設計】推論基盤のアーキテクチャ選定

### サマリー
本章の目的は、第2章で算出した理論上の目標値を実現するため、推論基盤を設計することにある。ハードウェアの性能を最大限に引き出すには、Triton Inference Serverのような高度な配信基盤を軸に、プロジェクトの性質に応じてvLLMやTensorRT-LLMをバックエンドとして選択する戦略が一般的である。本章では、これらの組み合わせによる必然的なアーキテクチャの詳細設計プロセスを提示する。

---

## 1. 推論配信基盤の選択：Triton Inference ServerとvLLM（APIサーバーモード）
細設計において、推論環境は単なるモデルの実行環境ではなく、ネットワーク越しにリクエストを制御する配信基盤として定義する。

FlaskやFastAPIを用いて自作のWebAPIサーバーを構築することも可能だが、汎用フレームワークではLLM特有のキューイング管理やGPUリソースの最大活用を自前で実装しなければならず、ハードウェアが遊んでしまうリスクが高い。これを回避するため、以下のいずれかの基盤を採用する。

* **Triton Inference Server**
  NVIDIAが提供する多機能な配信基盤である。複数のバックエンドを統合管理でき、gRPC通信やモデルアンサンブルなど、大規模かつ複雑な商用運用に耐えうる柔軟性を持つ。
* **vLLM（APIサーバーモード）**
  vLLMに内蔵されたサーバー機能を直接利用する形態である。Triton Inference Serverほど多機能ではないが、OpenAI互換のAPIを即座に提供できるため、LLM単体での推論配信においては最も軽量かつ効率的な選択肢となる。


### 2. バックエンドエンジンの論理的選択
Triton Inference Serverの背後で動く推論エンジンの選択が、限られたGPUリソースをどれだけ無駄なく使い切り、同時に処理できるリクエスト数をどこまで増やせるかを決定づける。

### Triton Inference Server + vLLM バックエンド：開発効率と柔軟性の両立
vLLMをバックエンドとして組み込む構成である。
* **PagedAttention機能**: OSの仮想メモリのようにKVキャッシュを断片化させずに管理する。これにより、メモリの空き容量を最大限に活用してバッチサイズを拡大し、一度に処理できるユーザー数を物理的に増大させる。
* **モデル対応の即時性**: Pythonベースの柔軟性を活かし、最新のオープンソースモデルを改修なしで即座にAPI化できる。最新技術の検証や、頻繁なモデル刷新が求められる環境において、インフラの機動力を担保する。

### Triton Inference Server + TensorRT-LLM バックエンド：演算性能の極限追求
NVIDIAのハードウェアに特化したTensorRT-LLMをバックエンドに採用する構成である。
* **カーネル最適化と量子化機能**: NVIDIA GPUの演算器（Tensor Core）に最適化された計算処理を行う。FP8等の最新の量子化技術を極限まで活用することで、計算時間を短縮し、圧倒的な低レイテンシを実現する。
* **垂直統合型ビルド**: 事前にモデルを特定のGPUに最適化したエンジンファイルへとコンパイルする工程が必要だが、その分、ハードウェアが持つ理論上の限界性能を限界まで引き出す。大規模・固定モデルの運用において、1トークンあたりの計算コストを最小化する。

---

#### Q3: 推論環境の設計において、ハードウェア性能を最大化するためのバックエンド採択基準は何か？

**回答の論理構成例：**
スループットという単一の指標に固執せず、運用コストや柔軟性を含めたトータルバランスを基準に評価する。

1. **基盤採用の判断**
   自作APIの限界を認識し、Triton Inference ServerやvLLM（APIサーバーモード）などの専用基盤を採用することで、GPUリソースの最大活用を担保する。
2. **検証フェーズ（vLLM）**
   モデル選定が流動的な段階では、開発の機動力を重視する。vLLMの柔軟性を活かし、SLAを維持できる構成を迅速に立ち上げる。
3. **本番運用フェーズ（TensorRT-LLM）**

# 第4章：【実装・最適化】動的メモリ管理とI/Oパイプラインの非同期設計

## サマリー
本章の目的は、計算リソースが確保されているにもかかわらず、システムが停止、あるいは低速化する実装上のボトルネックを排除することにある。第3章で選定した推論エンジンが持つポテンシャルを物理的な限界まで引き出すため、KVキャッシュの占有予測に基づく動的制御と、前処理・データ転送の遅延を演算の裏側で隠蔽するI/Oパイプラインの非同期設計手法を詳解する。

---

## 1. 実行時GPUメモリ管理の最適
理論上は収まるはずのGPUメモリが、実行時にOut of Memory（OOM）を起こす主因は、推論中に動的に増大するKVキャッシュの予約領域にある。

* **PagedAttention によるブロック管理**
  メモリを OS の仮想メモリ（ページ管理）のように、一定のトークン数を 1 ブロック（例：16 トークン）とした単位で管理します。各リクエストに対して事前に巨大な領域を予約せず、実際の生成トークン数に応じてリアルタイムで物理メモリ（ブロック）を割り当てることで、内部断片化を最小限に抑え、物理的なGPUメモリ容量を超える同時実行数を実現します。

> [!IMPORTANT]
> この「ブロック」は、OS のメモリ管理における 4KB といった物理的な容量ではなく、**論理的なトークンの数**で定義されます。これにより、モデルのパラメータ数や構造が異なっても、エンジニアは共通の尺度でメモリ消費を管理・予測できます。
* **KVキャッシュ・スワッピングの導入**
  GPUメモリが枯渇しそうになった際、進行中のリクエストを破棄するのではなく、一部のキャッシュ（メモリブロック）を一時的にホストメモリ（CPU側）へ退避させる。これにより、突発的な長文回答リクエストが混入しても、システム全体を停止させない弾力的な実装を行う。

---

## 2. I/Oパイプラインの非同期設計：GPU Starvationの解消
GPUの演算速度に対し、CPU側での処理や物理的なデータ転送がボトルネックとなる状態を、推論サーバー主導のプリフェッチとデータの集約によって解消する。

* **推論サーバーによる非同期制御とプリフェッチ**
  演算プロセスとは別に、ホスト側でデータ準備専用のスレッドを稼働させ、GPUの演算器を常にデータで満たし続けるラインを構築する。
    * **非同期コピーの活用**: PyTorch の `non_blocking=True` 指定や推論エンジンの内部スケジューラを通じ、データ転送の完了を待たずに次の CPU 処理（デコードやリクエストの集約）へ移行する設計を確立する。
* **データ形式の最適化とシリアライズの非同期化**
  JSON 等のテキスト形式ではなく、バイナリ形式（TFRecord や Protocol Buffers）を採用することで、CPU によるデータ解析（デシリアライズ）のオーバーヘッドを最小化する。
    * **オーバーラップ設計**: GPU が演算を実行している裏側で、ホスト側（CPU）が次順のデータをバイナリ化して準備しておくパイプラインを構築し、演算器へのデータ供給を途切れさせない。

> [!IMPORTANT]
> TFRecordは、大量のデータを一つのバイナリファイルにまとめる、高速読み込み専用のデータ形式である。数百万の小さなファイルを個別に出し入れする際のOS負荷（I/Oボトルネック）を排除し、ストレージ帯域を最大限に活用可能。データはシリアライズされた状態で保持されるため、CPUによる変換負荷が低く、GPU演算の裏側で次順のデータを準備する非同期パイプラインの構築に極めて有効な技術である。

* **推論エンジンの最適化設定**
  利用するフレームワークの特性に合わせ、GPU の演算密度を最大化するパラメーター調整を行う。
    * **並列実行数の制御（Triton: `instance_group`）**: 1 つの GPU 内で複数の推論インスタンスを並列稼働させ、演算コアの空き時間を最小化する。
    * **バッチ密度の最大化（vLLM: `max_num_batched_tokens`）**: 1 回の演算サイクルに投入する合計トークン数を調整し、スループット（効率）とレイテンシ（応答性）の最適なバランスを確保する。
---

#### Q4：モデルが理論上はGPUメモリに収まる計算であるにもかかわらず、実行時に CUDA Out of Memory (OOM) が発生したり、期待したスループットが出ない場合、どの設計不備を疑うか？ 同時に、高トラフィックな推論アーキテクチャにおいて、GPU Starvationを回避するために前後処理レイヤーをどのように設計すべきか？

**回答の論理構成例：**
要件定義に基づく静的なリソース見積もりの正当性を確認するだけでなく、実行時のメモリ予約の粒度とデータ供給の連続性という動的な挙動を基準に再点検を行う。

1. **メモリ使用状況の検証**
   PagedAttention を有効にしている場合でも、1ブロックあたりのトークン数設定が不適切であれば、ブロック内に大きな未使用領域が残る内部断片化が生じる。実行時の平均トークン消費傾向を分析し、物理的な空きメモリを使い切るために設定を最適化する。
2. **推論エンジンの並列・密度設定の妥当性**
   GPUメモリ容量に余裕があるにもかかわらずスループットが低い場合、`instance_group`（Triton）の設定不足により GPU の演算コアを遊ばせているか、あるいは `max_num_batched_tokens`（vLLM）が小さすぎて演算密度が上がっていない可能性がある。スループットとレイテンシのトレードオフを考慮しつつ、ハードウェアを使い切る設定値へ調整する。
3. **データ供給ラインの停滞（GPU Starvation）の特定**
   NVIDIA Nsight Systems 等のプロファイラーを用い、時間軸上で演算と転送がオーバーラップしているかを検証する。GPU の演算列に空白（アイドル時間）が生じ、その直前で CPU 側の前後処理やメモリ転送の完了を待っている状態であれば、非同期パイプラインの設計不備と判断し、プリフェッチやバッチ集約の最適化を行う。上位ライブラリの非同期転送フラグ（`non_blocking=True` 等）を適切に使用し、現在のバッチを計算している間に次順のバッチを GPUメモリへ送り込む計算と通信のオーバーラップが実現されているかを点検する。
4. **前後処理の非同期デカップリング**
トークナイズ等の CPU 負荷の高い処理が GPU ストリームを直接ブロックしないよう、前後処理を演算プロセスから論理的に分離する。また、JSON 等の解析負荷を避けるため TFRecord 等のバイナリ形式を採用し、演算の裏側で次順のデータ準備が完結する非同期構造（オーバーラップ）を設計する。

# 第5章：【コスト最適化】精度を損なわずにTCOを下げる量子化技術

### サマリー
本章の目的は、スケールに伴い増大するインフラコストへ対応策として、量子化技術によるリソース最適化を詳解することにある。AWQ や GPTQ 等の手法選択が、精度維持と計算効率のバランス、さらにはハードウェア寿命に与えるインパクトを評価する。単なるメモリ圧縮に留まらず、KVキャッシュの効率化を含めたスループット向上と、将来的な TCO（総保有コスト）削減を両立させるための導入判断基準を提示する。

---

### 1. 量子化によるリソース制約の再定義
量子化の最大の恩恵は、モデルの重みを 16bit から 4bit や 8bit へ圧縮することで、高価な GPU の必要枚数を物理的に削減できる点にある。

* **メモリフットプリントの劇的圧縮**
  70B クラスのモデルを FP16 で運用する場合、2枚以上の H100（160GB）が必須となるが、4bit 量子化（AWQ等）を施すことで約 40GB 程度に収まり、シングル GPU での運用が可能となる。
* **計算効率とメモリ帯域のトレードオフ**
  量子化はメモリ帯域制限（Memory-bound）下にある推論処理において、データ転送量を減らすことでスループットを向上させる。計算時のデコード処理のオーバーヘッドはあるものの、バッチサイズを拡大できるため、高トラフィック環境下では実効スループットが劇的に改善する。

### 2. KVキャッシュの圧縮と持続的な運用
モデルの重みだけでなく、推論中に生成される KVキャッシュの増大をいかに制御するかが、長期運用の安定性を左右する。

* **KVキャッシュ量子化（INT8/FP8）の導入**
  入力トークンが長くなるにつれ、GPUメモリの大半を KVキャッシュが占有するようになる。重みと同様に KVキャッシュも INT8 等で量子化することで、精度を維持したままメモリ消費量を半減させ、より長いコンテキストや大規模な同時リクエストへの対応を可能にする。
* **持続的なスケーラビリティの確保**
  キャッシュ領域を圧縮することで、GPUメモリのスワップ（CPU側への退避）の頻度を下げ、推論レイテンシのスパイクを抑制する。これは、ユーザー数やデータ量が増大し続ける実運用フェーズにおいて、インフラの追加投資を最小限に抑えつつ安定したサービス品質を維持するための必須技術である。

### 3. 量子化手法の選択と戦略的インパクト
プロジェクトの特性により、採用すべき手法は異なる。

* **AWQ (Activation-aware Weight Quantization)**
  重要な重みを保護しながら量子化する。特にチャット等の汎用タスクにおいて精度劣化が極めて少なく、現在の商用環境で最も使われる手法である。
* **GPTQ (Post-Training Quantization)**
  広く普及している手法。エコシステムが充実しており汎用性が高いが、極端な低ビット化では精度維持が難しくなる傾向がある。

---

#### Q5：顧客の予算が限られている一方で、LLM 推論において高いスループットを要求された場合、どのようなコスト最適化戦略を提案すべきか？

**回答の論理構成例：**
単なる安価なGPUへの変更ではなく、量子化技術を軸に、ハードウェアの利用効率を極限まで高める戦略を提示する。

1. **モデルと KVキャッシュの量子化によるリソース集約**
   モデルの重み（AWQ 等）と KVキャッシュの両方に量子化を適用することを提案する。これによりGPUメモリの消費を抑え、複数枚必要だった GPU を集約し、初期投資を削減する。
2. **スループット最大化によるユニットコストの低減**
   GPUメモリの節約分をバッチサイズの拡大に充て、1回あたりの演算効率を最大化する。結果として、1GPU あたりの同時処理リクエスト数を向上させ、1トークンあたりの単価を最小化する。
3. **持続可能な運用を見据えた拡張性**
   KVキャッシュ圧縮により、将来的な長文入力やトラフィック増大に対しても、現在のインフラ構成のまま対応可能であることを強調する。これは突発的なコスト増を防ぎ、長期的な TCO 削減に寄与する。
